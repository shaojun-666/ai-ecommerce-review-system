## 1. 项目概述

### 1.1 项目背景

电子商务行业快速发展，用户评论数据爆炸式增长。海量评论中蕴含的用户情感倾向、产品质量反馈、服务体验评价等信息，对商家改进产品和消费者购买决策都具有重要价值。然而，人工分析海量评论效率低下，传统关键词匹配方法无法理解语义和上下文。

**行业痛点：**
- 现有情感分析 API（阿里云、百度 AI、腾讯云）仅提供粗粒度的正/负/中分类，缺乏细粒度维度分析和可行动洞察
- 虚假评论（刷单、恶意差评）严重干扰真实评价体系，商家和消费者都无法有效鉴别
- 跨平台数据分散（京东、淘宝、拼多多），缺乏统一的智能化分析入口
- 通用大模型（如 ChatGPT）可以分析单条评论，但不支持批量处理、趋势追踪和结构化报表

### 1.2 项目目标

构建一个**AI驱动的电商评论智能分析系统**，在现有 API 和通用大模型之外提供差异化价值：

- 基于**蒸馏大语言模型 (Distilled LLM)** 的细粒度情感分析 — 比传统 BERT 更准确，零样本即可部署
- **虚假评论检测** — 基于多维度特征（文本异常、评分偏差、时间模式）识别可疑评论
- 跨平台评论对比与趋势分析
- 自动化评论摘要与可行动洞察生成
- 可视化数据看板展示分析结果

### 1.3 技术路线

```
后端框架: Flask + Gunicorn + gevent (异步支持)
前端框架: Vue 3 + Element Plus + ECharts
数据处理: Pandas + NumPy
AI 模型: Qwen2.5-1.5B (蒸馏) / BERT-base-chinese (基线) — 双轨对比
         + 异常检测模型 (Isolation Forest + 文本特征)
数据库: PostgreSQL 15 (业务) + Redis 7 (缓存 + Celery Broker)
部署: Docker Compose (Nginx + Gunicorn + Celery + PostgreSQL + Redis)
任务队列: Celery + Redis (异步批处理)
```

### 1.4 核心创新点

1. **蒸馏 LLM 情感分析 (2026 前沿)**：使用 Qwen2.5-1.5B 蒸馏模型替代传统 BERT，零样本准确率更高，且支持开放式方面提取（不限 4 个固定维度）。同时保留 BERT 作为基线对比，展示不同路线的性能差异。
2. **虚假评论检测引擎**：融合文本特征（语言学异常）、行为特征（评分时间模式）和统计特征（评分偏离度），使用集成学习方法识别虚假评论 — 这是现有商业 API 无法提供的功能。
3. **跨平台多维对比**：支持同一商品在京东、淘宝、拼多多等多平台的评论对比分析，帮助商家定位平台间差异。
4. **可行动洞察引擎**：不满足于"展示数据"，自动从负面评论中提取高频问题，生成按优先级排序的改进建议清单。

## 2. 需求分析

### 2.1 功能需求

| 编号 | 模块 | 功能描述 | 优先级 |
|------|------|---------|--------|
| F1 | 评论采集 | 支持 CSV/Excel 批量导入和 API 实时接入评论数据 | P0 |
| F2 | 情感分析 | 对每条评论进行正面/负面/中性分类，支持批量处理 | P0 |
| F3 | 细粒度分析 | 按产品质量、物流、服务、价格等维度分析情感 | P1 |
| F4 | 关键词提取 | 自动提取评论中的核心关键词和短语 | P1 |
| F5 | 评论摘要 | 对大量评论生成概括性摘要（正面总结 + 负面总结） | P1 |
| F6 | 趋势分析 | 展示情感趋势随时间变化（日/周/月） | P0 |
| F7 | 数据看板 | 可视化展示分析结果：图表、仪表盘、数据表格 | P0 |
| F8 | 商品对比 | 支持多商品/多店铺的评论对比分析 | P2 |
| F9 | 报告导出 | 支持分析报告导出为 PDF/Excel | P2 |
| F10 | 用户管理 | 多用户登录、权限管理（管理员/普通用户） | P1 |
| F11 | 数据管理 | 评论数据的增删改查、批量操作 | P1 |
| F12 | 预警通知 | 负面评论激增时触发预警通知 | P2 |

### 2.2 非功能需求

| 编号 | 需求 | 指标 |
|------|------|------|
| NF1 | 准确性 | 情感分析准确率 >= 85% |
| NF2 | 性能 | 单条评论分析 < 500ms，批量 1000 条 < 5min |
| NF3 | 并发 | 支持 50 并发请求 |
| NF4 | 可用性 | 系统可用性 >= 99% |
| NF5 | 安全性 | 用户认证、数据加密、防 SQL 注入 |
| NF6 | 可扩展性 | 支持水平扩展，新增分析维度无需重构 |

### 3.2 技术选型与理由

| 组件 | 选型 | 理由 |
|------|------|------|
| 后端框架 | Flask | 轻量灵活，适合 API 服务；如需全功能选择 Django |

| UI 组件库 | Element Plus | 成熟稳定，中文文档完善 |
| 可视化 | ECharts | 图表类型丰富，交互性好 |
| NLP 框架 | PyTorch + Transformers | BERT 生态最完善 |
| 模型优化 | ONNX Runtime | 推理速度提升 2-4 倍 |
| 数据库 | PostgreSQL | 支持 JSON 字段，适合分析结果存储 |
| 缓存 | Redis | 高性能缓存 + 任务队列 |
| 任务队列 | Celery + Redis | 异步处理批量分析任务 |
| 部署 | Docker Compose | 一站式部署，环境一致 |

-- 评论表

    id              SERIAL PRIMARY KEY,

## 7. NLP 模型设计

### 7.1 数据策略

#### 数据集来源

| 数据集 | 规模 | 用途 | 获取方式 |
|--------|------|------|---------|
| 京东评论数据集 (公开) | 50,000+ 条 | 训练 + 评估 | 已标注，直接使用 |
| 淘宝评论数据集 (公开) | 30,000+ 条 | 训练 + 评估 | 已标注，直接使用 |
| 人工标注验证集 | 500 条 | BERT 微调 + 双模型评估 | 团队标注 (非关键路径，Week 1 启动) |

**重要变更：使用蒸馏 LLM 后标注需求从 5000 条降至 500 条** — 零样本能力大幅降低了对标注数据的依赖。

#### 情感标签

| 标签 | 说明 | 示例 |
|------|------|------|
| positive | 正面评价 | "这个产品质量很好，非常满意" |
| negative | 负面评价 | "发货太慢了，差评" |
| neutral | 中性/客观 | "价格适中，用了几天再看看" |

#### 方面标签 (Aspect) — 开放式

蒸馏 LLM 不限制固定维度，模型动态提取。以下为常见方面的参考列表：

| 方面 | 说明 | 典型关键词 |
|------|------|-----------|
| quality | 产品质量 | 质量、做工、耐用、材质、手感 |
| logistics | 物流配送 | 发货、物流、快递、配送、包装 |
| service | 客服服务 | 客服、售后、退换货、态度 |
| price | 价格性价比 | 价格、性价比、值、划算、贵 |

#### 虚假评论检测特征

| 特征类别 | 具体特征 | 维度数 |
|----------|---------|-------|
| 文本特征 | 语言模型困惑度、TF-IDF 异常度、重复度、情感-评分矛盾 | 8 |
| 行为特征 | 评论时间间隔、批量发布模式、账号活跃度、IP 聚集度 | 8 |
| 统计特征 | 评分偏离度、字数异常、极端评分比例、历史模式 | 8 |

### 7.2 双轨模型路线

#### 路线 A: 蒸馏 LLM (Qwen2.5-1.5B + LoRA) — 主力模型

```
Input: 这个手机电池续航很不错，但发货有点慢
                │
                ▼
        Qwen2.5 Tokenizer
                │
                ▼
    Qwen2.5-1.5B Decoder (28层)
      ┌─────────────────────────┐
      │  LoRA Adapters (rank=8) │
      │  (冻结原权重，只训练适配器)│
      └───────────┬─────────────┘
                  │
                  ▼
    Prompt 模板 → 结构化 JSON 输出
    {
      "sentiment": "positive",
      "aspects": ["quality", "logistics"],
      "is_fake": false,
      "keywords": ["续航", "发货"],
      "confidence": 0.92,
      "summary": "产品本身质量受认可，物流体验有待改进"
    }
```

**优势：** 零样本部署、一个模型完成所有任务、开放式方面提取
**代价：** 推理速度较慢 (~300ms/条)、显存需求较高 (~1.5GB 4bit 量化后)

#### 路线 B: BERT-base-chinese — 基线对照

```
Input: 这个手机电池续航很不错
                │
                ▼
        BERT Tokenizer → Encoder → [CLS]
                │
                ▼
    ┌──────────────────────┐
    │  多任务分类头         │
    │  - 情感 (3类)        │
    │  - 方面 (4类)        │
    └──────────────────────┘
```

**优势：** 推理速度快 (~80ms/条)、部署轻量 (~400MB)
**代价：** 需要标注数据、仅限固定 4 维度、无法检测虚假评论

#### 双路线对比实验

| 维度 | BERT | Qwen2.5-1.5B (LoRA) |
|------|------|---------------------|
| 情感分类 F1 | 基线 | 预期 +3-5% |
| 方面提取 | 4 固定维度 | 开放式 |
| 虚假评论检测 | 不支持 | 支持 |
| 推理速度 | ~80ms | ~300ms (需量化) |
| 训练数据 | 5000+ 条 | 0-500 条 |
| 部署显存 | ~400MB | ~1.5GB (4bit) |

**预期结论：** 蒸馏 LLM 在准确率、灵活性方面显著优于 BERT，但推理速度较慢。项目将给出明确的选型建议。

### 7.3 训练流程

```
Phase 1: 数据准备 (Week 1-2)
  ├── 收集公开数据集 (已标注，直接使用)
  ├── 数据清洗和预处理 (去除 HTML、特殊字符、标准化)
  ├── 人工标注 500 条验证集 (非关键路径，Week 1 即可启动)
  └── 简单文本增强 (随机掩码、同义词替换，非回译)

Phase 2: BERT 基线训练 (Week 3-4)
  ├── 加载 bert-base-chinese 预训练权重
  ├── 添加多任务分类头 (情感 3 类 + 方面 4 类)
  ├── 训练: batch_size=32, lr=2e-5, epochs=5
  ├── 评估指标: Accuracy, Precision, Recall, F1
  └── 模型保存 + ONNX 导出

Phase 3: 蒸馏 LLM 训练 (Week 4-5)
  ├── 加载 Qwen2.5-1.5B-Instruct + 4bit 量化
  ├── LoRA 微调 (rank=8, 500 条标注数据)
  ├── 评估 + Bad Case 分析
  └── 对比 BERT vs LLM → 输出选型报告

Phase 4: 虚假评论检测 (Week 5-6)
  ├── 特征工程 (24 维特征)
  ├── Isolation Forest + XGBoost 集成
  ├── 在合成数据上评估
  └── 集成到推理管线

Phase 5: 优化部署 (Week 6-7)
  ├── ONNX / vLLM 导出优化
  ├── 推理性能基准测试
  ├── 缓存层集成 (Redis)
  └── 集成到后端 API
```

### 7.4 评估指标

| 指标 | 目标值 (BERT) | 目标值 (LLM) | 计算方法 |
|------|-------------|-------------|---------|
| 情感分类 Accuracy | >= 85% | >= 88% | 正确预测 / 总样本数 |
| 方面提取 F1 | >= 80% | >= 85% | 2 * P * R / (P + R) |
| 虚假评论 AUC | N/A | >= 0.80 | ROC AUC |
| 推理速度 | < 100ms | < 500ms | 单条推理，CPU |
| 零样本冷启动 | 不支持 | 开箱即用 | 无训练数据下的可用性 |

### 7.5 ONNX 推理优化

```python
# BERT 推理性能
# PyTorch: ~150ms/条 → ONNX FP32: ~80ms/条 (1.9x)

# LLM 推理性能 (Qwen2.5-1.5B)
# PyTorch FP16: ~500ms/条 → vLLM: ~200ms/条
# 4bit AWQ 量化 + vLLM: ~150ms/条

# 策略：BERT 用于高吞吐量场景 (批量分析)，LLM 用于高质量场景 (单条详细分析)
```

- [ ] **D1: 项目初始化** (CC: 1h)

  - 添加缺失索引 (idx_tasks_user_status, idx_results_task, idx_comments_product_created 等)

  - 权限中间件 + 审计日志
  - **限流保护** (Flask-Limiter: 登录 5次/分钟/IP, 注册 2次/小时/IP)

#### Phase 2: 后端核心功能 (Week 3-5) — CC: ~15 小时 / Human: ~20 小时

- [ ] **D5: 商品管理 API** (CC: 1h)
  - CRUD 接口 + 数据验证
  - 分页/排序/筛选
- [ ] **D6: 评论管理 API** (CC: 3h / Human: 4h)
  - 导入接口：CSV/Excel 流式解析 (非一次性加载)
  - **编码检测**：chardet 自动检测 GBK/UTF-8/GB18030
  - **文件大小限制**：MAX_CONTENT_LENGTH = 50MB
  - **行级错误报告**：单行失败不中断导入
  - 列表接口 (筛选/排序/分页 + 复合索引优化)
  - 详情/删除接口 (并发删除保护)
  - 数据导出接口 (CSV sanitization 防公式注入)
- [ ] **D7: 分析任务系统** (CC: 4h / Human: 5h)
  - 任务创建/状态查询
  - Celery 异步执行 (acks_late=True, reject_on_worker_lost=True)
  - **任务进度推送给前端**：方案 A: HTTP 长轮询 (默认，兼容性好) / 方案 B: Flask-SocketIO + gevent (可选)
  - **心跳 + 超时处理**：Celery Beat 定时检查超时任务 → 自动标记 failed
  - **部分失败处理**：`completed_with_errors` 状态 + error_count 计数
- [ ] **D8: 看板数据 API** (CC: 3h / Human: 4h)
  - 总览聚合查询 (复合索引优化 + EXPLAIN ANALYZE 验证)
  - **Redis 缓存策略**：
    - `dashboard:overview:{user_id}` → TTL 5min
    - `dashboard:trend:{product_id}:{period}` → TTL 10min
    - 分析完成时主动失效相关缓存
  - 趋势时间序列

  - **虚假评论看板**：高 fake_score 评论列表 + 可疑模式突出显示

## 11. 部署方案

### 11.1 Docker Compose 架构

```yaml
version: '3.8'
services:
  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf
      - ./frontend/dist:/usr/share/nginx/html
    depends_on:
      - backend

  backend:
    build: ./backend
    environment:
      - FLASK_ENV=production
      - DATABASE_URL=postgresql://user:pass@db:5432/ecommerce_ai
      - REDIS_URL=redis://redis:6379/0
    depends_on:
      - db
      - redis
    volumes:
      - model_data:/app/nlp/models

  celery_worker:
    build: ./backend
    command: celery -A app.tasks worker -l info
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/ecommerce_ai
      - REDIS_URL=redis://redis:6379/0
    depends_on:
      - backend
      - redis

  db:
    image: postgres:15-alpine
    environment:
      - POSTGRES_DB=ecommerce_ai
      - POSTGRES_USER=user
      - POSTGRES_PASSWORD=pass
    volumes:
      - pgdata:/var/lib/postgresql/data

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"

volumes:
  pgdata:
  model_data:
```

## 12. 风险评估与应对

### 12.1 技术风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| BERT 模型准确率不达标 | 中 | 高 | 准备多套预训练模型候选 (RoBERTa/ALBERT)，预留调优时间 |

| 批量分析性能瓶颈 | 低 | 中 | 异步任务 + 结果缓存，水平扩展 Celery Worker |

| 风险 | 概率 | 影响 | 应对措施 |
