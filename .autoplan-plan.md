## 一、亟需修正的问题 (必须先修后进)

### 🔴 Bug 1: 后端测试因 SQLite 池配置全部失败

- **症状**: `test_api.py`、`test_auth.py`、`test_models.py` 中 ~23 个测试全部报 `sqlite3.OperationalError` 池大小错误
- **根因**: `backend/app/config/__init__.py` 中 `TestingConfig` 继承自 `Config`，`SQLALCHEMY_ENGINE_OPTIONS = {"pool_size": 10, ...}` 对 SQLite 无效
- **修复**: TestingConfig 中覆盖 `SQLALCHEMY_ENGINE_OPTIONS = {}` 或用 `poolclass=NullPool`
- **影响**: 无法通过 CI 验证后端变更，每次修改都需手动测
- **工作量**: 5min CC / 10min Human
- **建议**: 立即修

### 🔴 Bug 2: 前端 3 个 API 端点后端不存在

| 前端页面 | 调用的 API | 后端状态 | 影响 |
|---------|-----------|---------|------|
| `Register.vue` | `POST /auth/register` | 不存在 | 用户无法注册 (登录可工作) |
| `Settings.vue:updateProfile` | `PUT /users/me` | 不存在 | 个人设置页修改资料不可用 |
| `Settings.vue:updatePassword` | `PUT /users/me/password` | 不存在 | 密码修改不可用 |

- **修复**: 在 `backend/app/api/v1/auth.py` 增加 register 端点; 在 `users.py` 增加 `/users/me` 和 `/users/me/password`
- **影响**: 3个前端页面功能不可用
- **工作量**: 15min CC / 30min Human
- **建议**: 立即修

### 🟡 Bug 3: 看板缓存未在爬虫完成后失效

- **症状**: 爬虫采集新评论后，Dashboard 数据仍显示旧数据 (缓存 TTL 5min)
- **根因**: `backend/app/services/analysis_service.py` 中的 `invalidate_dashboard_cache()` 只在分析任务完成时调用，爬虫任务不触发
- **修复**: `crawl_tasks.py` 的 `run_crawl` 成功后调用 `invalidate_dashboard_cache()`
- **影响**: 爬虫完成后最多等5分钟才能看到新数据
- **工作量**: 5min CC / 5min Human
- **建议**: 立即修

### 🟡 Bug 4: NLP Dockerfile 未集成到 docker-compose

- **症状**: `nlp/Dockerfile` (pytorch/pytorch:2.1.0-cuda12.1, ~7GB) 存在但未被 docker-compose.yml 引用。NLP 模型实际运行在 backend 容器内 (python:3.10-slim)
- **影响**: GPU 无法利用; 如离线训练场景需单独启动容器
- **修复**: 2个选择——(A) 保持现状，NLP推理走backend CPU; (B) 将 NLP 作为独立服务加入 compose，GPU 加速推理
- **工作量**: (A) 0 / (B) 1h CC / 2h Human
- **建议**: 近期选A、远期若需大规模推理选B

### 🟡 Bug 5: AnalysisList.vue 发送空 comment_ids

- **症状**: 创建分析任务时 `comment_ids: []` 为空数组，用户无法选择要分析的评论子集
- **根因**: 前端任务创建表单未实现评论选择器，固定传空数组
- **影响**: 分析任务只能分析全部评论，无法定向分析
- **修复**: 前端增加评论多选组件; 后端已在 `analyze_comments_task` 中处理筛选逻辑
- **工作量**: 20min CC / 30min Human
- **建议**: P1 修复

### 🟡 Bug 6: `reportlab==4.0.8` 在 requirements.txt 但未被使用

- **症状**: 依赖中存在 `reportlab`，但所有报告导出使用 CSV (report_service.py)，不涉及 PDF
- **影响**: 增加容器镜像大小和不必要的依赖
- **修复**: 从 requirements.txt 移除
- **工作量**: 1min
- **建议**: 清除

## 二、架构性问题

### 🔶 问题 1: 项目配置重复

- `backend/app/config/__init__.py` — 活跃配置 (3个环境类)
- `backend/app/config/default.py` — 旧配置 (部分覆盖、部分冲突)
- 建议: 统一到 `__init__.py`，删除 `default.py`

### 🔶 问题 2: 文本清洗代码重复

- `backend/app/utils/text_cleaner.py` ≈ `nlp/src/data_processing/cleaner.py`
- 两者功能高度重叠但不同步，改了一个另一个会不一致
- 建议: 将 NLP cleaner 作为唯一源，backend 通过 import（而非复制）使用

### 🔶 问题 3: 缺少 HTTPS

- nginx.conf 只监听 80 端口，无 SSL 配置
- 生产部署需要 Let's Encrypt 或反向代理 SSL 终结
- 建议: 生产部署时增加

### 🔶 问题 4: flask-limiter 使用内存存储

- 启动日志警告: 速率限制使用内存存储而非 Redis
- 多 worker 场景下不准确
- 建议: 配置 Redis 作为 limiter 存储后端

## 七、关键决策点

### D1: NLP 模型隔离策略

**问题**: NLP Dockerfile (CUDA 7GB) 未集成到 compose，推理在 backend CPU 跑

**选项**:
- A) **保持现状** (推荐) — CPU推理，NLP容器预留用于离线训练
  - 当前评论量级(百/千) CPU推理绰绰有余
  - 省去 GPU 容器管理和 CUDA 依赖复杂性
  - 等到日均处理万级评论再考虑 GPU
  
- B) **NLP 独立服务** — 将 NLP 作为 gRPC/REST 微服务
  - GPU 加速推理，适合大规模处理
  - 增加部署复杂性 + 额外容器
  - 需要 nvidia-docker 支持

**建议**: 选 A。当前系统规模不需要 GPU 推理。B 在需要批量处理数万评论 / 上线 LLM 全量分析时再考虑。

### D2: 多平台爬虫优先级

**问题**: 淘宝和拼多多爬虫的开发成本 / 收益

**选项**:
- A) **先做淘宝** (推荐)
  - 淘宝评论 API 相对公开，适配较快
  - 淘宝商品数据覆盖率高(种子数据中有淘宝商品)
  - 用户可能最需要
  
- B) **先做拼多多**
  - 拼多多增速快，价格优势明显
  - 反爬可能更强，适配难度高
  
- C) **京东深度优化后再扩展**
  - 先完善京东价格追踪、评论全量采集
  - 单平台做到极致

**建议**: 选 A，Phase 8 以淘宝为主，拼多多降为 Phase 8.5。

### D3: 上行期检测方案

**问题**: 如何定义和检测"上行期商品"

**建议方案**: 多因子评分模型
```
上行期评分 = w1 × 评论增长率(最近7天 vs 前7天)
           + w2 × 情感趋势(最近评论情感斜率)
           + w3 × 价格稳定性(价格波动方差)
           + w4 × 销量代理指标(评论增速/在架时间)
```

无需复杂的深度学习模型。基于规则的加权评分 + 简单统计模型即可在初期达到 80%+ 准确率。LLM 用于生成自然语言的选品建议，而非做数值预测。
