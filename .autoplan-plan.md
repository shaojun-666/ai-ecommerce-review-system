## 项目现状

前端(Vue 3 + Element Plus)、后端(Flask API)、NLP(BERT/Qwen2.5) 三端框架已搭建完成，132 项测试全部通过，Docker 部署 5/5 容器健康运行。

**当前缺失的核心能力：**
- 没有自动爬虫模块 — 数据仅支持 CSV/Excel 手动导入
- 没有数据过滤/去重管道
- 没有周期性数据采集调度
- 没有实时数据展示机制
- 没有增量模型训练流程

## 目标

在现有系统基础上，先提升 NLP 模型质量（利用已有80k+数据集），再构建京东单平台爬虫，实现"模型驱动+数据辅助"的差异化策略。

## 实施策略（基于CEO审核调整）

### 策略调整

根据 CEO 审核反馈，做出以下关键调整：

1. **优先级反转**：先提升模型质量（用现有数据集达到 90%+ 准确率），再构建爬虫
2. **平台精简**：仅支持京东（最易对接，现有数据集可验证），淘宝/拼多多 → P2
3. **避免过度工程**：实时 = 30s HTTP 轮询，WebSocket → P1
4. **复用现有基础设施**：Celery Beat 替代 APScheduler，复用现有 NLP cleaner
5. **去重方案**：MinHash 字符级 n-grams（2-3grams），替代 SimHash

## Key Engineering Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | 爬虫为Celery task | 避免阻塞Flask worker，复用Celery Beat调度 |
| 2 | 暂存区表(staging) | 解耦采集和过滤，支持分阶段重试 |
| 3 | 独立CrawlTask模型 | AnalysisTask状态机不匹配爬虫生命周期 |
| 4 | requests+BS4非Scrapy | 轻量无额外依赖，京东API走XHR非渲染 |
| 5 | 响应分类器(anti-bot) | 检测CAPTCHA/login wall/WAF blocks |
| 6 | bleach消毒HTML | 防止爬取评论中的XSS攻击 |
| 7 | URL白名单(https+jd.com) | 防止SSRF和URL注入 |
| 8 | 双缓冲模型热加载 | 防止写时读竞争导致推理崩溃 |
| 9 | MinHash+LSH预计算 | 避免O(n*m)全量比较 |
| 10 | 动态batch size(至500) | 提升大批量写入性能 |

## 核心需求（调整后）

### P0 (必须实现)
1. **NLP 模型质量提升** — 用现有 80k+ 数据集将 BERT 情感分类准确率从 85% 提升至 90%+
   - 数据增强（回译、同义词替换）
   - 超参数调优（学习率、batch size、warmup）
   - 错误分析 + Bad Case 驱动迭代
   - ONNX 推理优化
2. **京东爬虫引擎** — 仅京东平台的商品信息/评论爬取
   - 遵循 robots.txt + 请求频率控制（1-3秒/请求）
   - 随机 User-Agent 轮换
   - 结构化数据提取（商品名、价格、评论内容、评分、时间）
   - 爬虫健康监控（最后成功时间、错误率）
3. **爬虫调度器** — Celery Beat 定时/按需启动采集任务
   - 任务状态跟踪（复用 AnalysisTask 模型）
   - 可配置采集间隔（默认 6h）
4. **数据过滤管道** — 爬取后的数据清洗与去重
   - MinHash 字符级 n-gram 去重
   - 正则规则垃圾评论过滤
   - 复用现有 NLP cleaner 标准化
5. **数据入库** — 过滤后的数据写入现有 Comment/Product 表
   - Schema 验证（确保字段完整性）
   - 数据新鲜度标识（最后刷新时间）
6. **实时信息展示** — 30s HTTP 轮询更新
   - 最新评论流
   - 采集任务状态看板
   - 数据概览自动刷新

### P1 (重要)
7. **增量模型训练** — 利用新采集数据重新训练/微调模型
8. **数据源管理** — 添加/管理需要监控的商品链接
9. **告警机制** — 负面评论激增时自动告警
10. **WebSocket 推送** — 取代30s轮询（可选）

### P2 (锦上添花)
11. **多平台扩展** — 添加淘宝/拼多多适配器
12. **跨平台对比** — 同一商品不同平台评论对比
13. **定时报告** — 自动生成并发送分析报告

## Error & Rescue Registry

| 场景 | 错误类型 | 触发条件 | 捕获层 | 用户看到 | 已测试? |
|------|---------|---------|--------|---------|---------|
| 爬虫目标平台变更HTML | ExtractionError | 平台更新页面结构 | 爬虫适配器 | "数据采集失败" + 重试按钮 | 需添加 |
| 爬虫被限流 | RateLimitError | 请求频率过高触发 CAPTCHA | 爬虫中间件 | 自动降速 + 状态提示 | 需添加 |
| 爬虫目标页面404 | PageNotFoundError | 商品下架/链接失效 | 爬虫调度器 | "商品已不存在" + 删除建议 | 需添加 |
| 数据去重全量重复 | DuplicateError | 采集到与库中完全一致的数据 | 过滤管道 | 0条新增 + "数据已是最新" | 需添加 |
| NLP模型加载失败 | ModelLoadError | 模型文件损坏/路径变更 | SentimentService | "分析服务暂不可用" + 重试 | 已有测试 |
| CSV导入编码错误 | EncodingError | GBK/UTF-8自动检测失败 | 导入服务 | "文件编码不识别" + 手动选择 | 已有测试 |
| Celery任务超时 | TaskTimeoutError | 分析超过30分钟 | Celery Beat | "任务超时" + 重新分析 | 已有测试 |
| Redis不可用 | CacheUnavailable | Redis宕机 | 缓存层 | 走直连（无缓存） | 已有降级 |

## Phase 2: 详细任务分解 — 京东爬虫引擎

### ⚠️ 复杂性警示（来自ENG审核）
京东评论数据通过 XHR 动态加载（`club.jd.com/comment/productPageComments.action`），非服务器端渲染。不能简单用 `requests + BS4` 请求商品详情页。需要额外探针研究：
- 需分析 XHR 请求参数（callback, score, sortType, page）
- 需处理 Cookie/TLS JA3 指纹
- 高人气商品（10k+评论）单次采集需 100-300 秒
- 预算：原 5h CC / 7h Human → 调整为 8h CC / 15h Human

## Cross-Phase Themes

   - 对策: D8中新增安全消毒步骤，所有新API端点继承现有认证

   - 对策: Phase 2新增探针研究步骤，调整预算为8h CC / 15h Human

   - CEO: 无爬虫测试/监控策略
